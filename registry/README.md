# Replay 2026 Badge Community Apps Registry

This folder hosts the registry the badge fetches over WiFi to populate its
Community Apps screen. The registry lets us ship installable apps and
downloadable user files without rebuilding firmware.

Two files live here:

- [`community_apps.json`](community_apps.json) is schema v2, which current
  firmware reads. It is generated by
  [`firmware/scripts/generate_startup_files.py`](../firmware/scripts/generate_startup_files.py)
  from files under `registry/community_apps/`, plus any curated
  remote assets in `registry/registry.json`.
- [`registry.json`](registry.json) is schema v1, kept for older firmware that
  still reads the legacy asset list.

## How The Badge Consumes It

1. The firmware sets `community_apps_url` to
   `https://raw.githubusercontent.com/temporal-community/badge.temporal.io/main/registry/community_apps.json`.
2. When WiFi is connected, the badge fetches the registry and shows entries in
   Settings -> Community Apps.
3. The user picks an entry; the badge downloads the file or app bundle into a
   temporary path on FATFS, optionally checks SHA-256, and atomically renames it
   into place.

The legacy settings key `asset_registry_url` still maps to the same registry
URL internally.

## Release Model

The public release workflow uploads `replay2026-factory-16MB.bin`, a full
factory image that already includes `firmware/initial_filesystem/doom1.wad`.
The registry can still expose the WAD as a downloadable file for repair or
reinstall flows, but it points at the committed file in this repo rather than a
separate release attachment.

Future large optional assets can either be committed when licensing and size
allow or hosted on a stable HTTPS URL. See
[`release-assets/README.md`](../release-assets/README.md) for the firmware
release artifacts that are uploaded by the GitHub release workflow.

For how registry installs interact with FATFS, app0, and user files, see
[`firmware/docs/STORAGE-MODEL.md`](../firmware/docs/STORAGE-MODEL.md).

## Schema V2

Registry entries are either single files (`kind: "file"`) or multi-file apps
(`kind: "app"`).

File entries use:

- `id`: stable key, do not reuse.
- `name`: display name shown on the badge.
- `version`: opaque version string.
- `dest_path`: absolute FATFS destination.
- `url`: HTTPS download URL.
- `sha256`: optional corruption check.
- `size`: byte size for progress and free-space checks.
- `min_free_bytes`: optional install guard.
- `description`: short detail text.

App entries use `dest_dir` plus a `files` array of per-file paths, sizes,
hashes, and URLs. The registry inlines the app file list so a single fetch
covers the catalog.

## Updating Entries

For installable community apps that should not ship in the factory filesystem,
update `registry/community_apps/<app>/`. Then rerun:

```sh
python3 firmware/scripts/generate_startup_files.py
```

Then commit the generated `registry/community_apps.json` with the source file
changes.

For the legacy v1 registry, edit `registry.json` by hand and keep it focused on
backward-compatible single-file entries.

## Security Stance

The registry SHA-256 values are corruption checks, not signatures. Keep URLs
stable, keep JSON valid, and prefer atomic destination paths so interrupted
downloads do not leave partial files in place.
